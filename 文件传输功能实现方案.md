# 文件传输功能实现方案

## 1. 目标

本文档用于设计当前聊天室项目中的文件传输功能，采用简单、稳定、容易演示的方案：

```text
前端上传文件 -> 后端保存到本地磁盘 -> MySQL 保存文件元数据 -> WebSocket 发送文件消息 -> 接收方点击下载
```

本方案暂不接入 OSS，也不通过 WebSocket 直接传输文件二进制。WebSocket 只负责发送“文件消息”，文件本体通过 HTTP 上传/下载。

推荐文件存储目录：

```text
UserUploadFiles/
```

该目录放在项目主目录下，例如：

```text
/Users/bytedance/JavaProject/Graduation/UserUploadFiles
```

后续如果迁移到 OSS，只需要把本地存储路径替换为 OSS object key 或 URL，聊天消息协议可以基本不变。

## 2. 为什么不直接用 WebSocket 传二进制

虽然 Netty WebSocket 支持二进制帧，但对当前项目来说不建议一开始这样做，原因是：

- 接收方离线时，文件仍然需要落地保存。
- 大文件通过 WebSocket 转发容易带来内存压力。
- 断线、重传、进度、失败恢复会复杂很多。
- 历史消息展示时仍然需要一个文件下载入口。

因此更简单的闭环是：

```text
文件先上传并保存，聊天里只传 fileId
```

## 3. 总体交互流程

### 3.1 发送文件

```text
1. 用户在前端选择文件
2. 前端调用 POST /file/upload 上传文件
3. 后端把文件保存到 UserUploadFiles 目录
4. 后端在 MySQL 中保存文件元数据
5. 后端返回 fileId、fileName、fileSize、contentType
6. 前端通过 WebSocket 发送一条“文件消息”
7. 接收方收到 WebSocket 消息后展示文件卡片
```

### 3.2 下载文件

```text
1. 接收方点击文件卡片
2. 前端请求 GET /file/download/{fileId}
3. 后端校验当前用户是否有权限下载
4. 后端从本地磁盘读取文件并返回
```

## 4. 后端实现思路

### 4.1 新增本地文件目录

建议在项目主目录创建：

```text
UserUploadFiles/
```

目录结构可以按日期拆分，避免单目录文件过多：

```text
UserUploadFiles/
  2026/
    05/
      06/
        9f4d0c8b8d4f4dbb9d6a6f2f8d9e6a01.png
```

建议把该目录加入 `.gitignore`，不要把用户上传文件提交到 Git。

### 4.2 增加配置

可以在 `application.yml` 中增加：

```yaml
file:
  upload:
    base-dir: UserUploadFiles
    max-size-mb: 10
```

说明：

- `base-dir` 使用相对路径时，默认相对于项目运行目录。
- 本地 IDEA 启动时，通常是项目主目录。
- 如果部署到服务器，可以改成绝对路径，例如 `/opt/graduation/UserUploadFiles`。

### 4.3 新增文件元数据表

建议新增一张表，例如 `graduation_upload_file`：

```sql
CREATE TABLE graduation_upload_file (
    file_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    uploader_id BIGINT NOT NULL,
    original_name VARCHAR(255) NOT NULL,
    stored_name VARCHAR(255) NOT NULL,
    storage_path VARCHAR(512) NOT NULL,
    content_type VARCHAR(128),
    file_size BIGINT NOT NULL,
    file_ext VARCHAR(32),
    create_time DATETIME NOT NULL,
    INDEX idx_uploader_id (uploader_id),
    INDEX idx_create_time (create_time)
);
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `file_id` | 文件 ID，前端发送文件消息时使用 |
| `uploader_id` | 上传用户 ID |
| `original_name` | 用户上传时的原始文件名 |
| `stored_name` | 后端生成的磁盘文件名，避免重名 |
| `storage_path` | 文件在本地磁盘的相对路径 |
| `content_type` | MIME 类型，例如 `image/png` |
| `file_size` | 文件大小，单位字节 |
| `file_ext` | 文件后缀 |
| `create_time` | 上传时间 |

### 4.4 新增实体与 Mapper

建议在 `Graduation-Common` 新增实体：

```text
Graduation-Common/src/main/java/com/gm/graduation/common/domain/UploadFile.java
```

字段对应 `graduation_upload_file` 表。

在 `IM-Bootstrap` 新增 Mapper：

```text
IM-Bootstrap/src/main/java/com/gm/imbootstrap/mapper/UploadFileMapper.java
```

继承 MyBatis Plus 的 `BaseMapper<UploadFile>` 即可。

### 4.5 新增 FileService

建议新增：

```text
IM-Bootstrap/src/main/java/com/gm/imbootstrap/service/FileService.java
```

主要方法：

```java
UploadFile upload(MultipartFile file, Long uploaderId);

UploadFile getFile(Long fileId);

Resource loadAsResource(Long fileId, Long currentUserId);
```

上传逻辑：

```text
1. 校验用户已登录
2. 校验文件不为空
3. 校验文件大小不超过 max-size-mb
4. 生成 UUID 文件名
5. 按日期创建目录
6. 保存文件到 UserUploadFiles/yyyy/MM/dd/
7. 插入 graduation_upload_file 元数据
8. 返回 fileId 等信息给前端
```

文件名处理建议：

- 原始文件名只用于展示，不直接作为磁盘文件名。
- 磁盘文件名使用 UUID，例如 `uuid + .png`。
- 后缀从原始文件名中取，注意只保留最后一个 `.` 后的部分。

### 4.6 新增 FileController

建议新增：

```text
IM-Bootstrap/src/main/java/com/gm/imbootstrap/controller/FileController.java
```

上传接口：

```text
POST /file/upload
Content-Type: multipart/form-data
Authorization: Bearer token
```

请求字段：

```text
file: MultipartFile
```

返回示例：

```json
{
  "code": 200,
  "message": "上传成功",
  "data": {
    "fileId": 10001,
    "fileName": "测试文件.pdf",
    "fileSize": 102400,
    "contentType": "application/pdf"
  }
}
```

下载接口：

```text
GET /file/download/{fileId}
Authorization: Bearer token
```

响应：

```text
文件二进制流
```

响应头建议：

```text
Content-Type: 文件 MIME 类型
Content-Disposition: attachment; filename="原始文件名"
Content-Length: 文件大小
```

### 4.7 下载权限控制

最简单做法：

- 上传者可以下载。
- 如果文件已经作为私聊消息发送，发送者和接收者可以下载。
- 如果文件已经作为聊天室消息发送，聊天室成员可以下载。

第一阶段为了简单，可以先做：

```text
登录用户可以下载自己上传的文件；
接收文件消息的一方下载权限后续再补。
```

但如果要做得更完整，建议新增一张文件消息关联表，或者从消息表中解析 `content` 中的 `fileId` 判断权限。

更推荐的简单做法是，在文件上传时不决定接收方权限，而是在发送文件消息后，通过消息记录来证明权限：

```text
private message content 包含 fileId
当前用户是 senderId 或 receiverId -> 允许下载
```

聊天室同理：

```text
chatroom message content 包含 fileId
当前用户属于该 room -> 允许下载
```

### 4.8 WebSocket 文件消息设计

WebSocket 不传文件本体，只传文件元数据 JSON。

私聊文件消息建议：

```json
{
  "appId": 2,
  "uid": 1,
  "toId": 2,
  "messageType": 2,
  "content": "{\"messageKind\":\"FILE\",\"fileId\":10001,\"fileName\":\"测试文件.pdf\",\"fileSize\":102400,\"contentType\":\"application/pdf\"}",
  "timeStamp": 1715000000000
}
```

聊天室文件消息建议：

```json
{
  "appId": 1,
  "uid": 1,
  "toId": 10,
  "messageType": 3,
  "content": "{\"messageKind\":\"FILE\",\"fileId\":10001,\"fileName\":\"测试文件.pdf\",\"fileSize\":102400,\"contentType\":\"application/pdf\"}",
  "timeStamp": 1715000000000
}
```

注意：

- `content` 仍然是字符串，里面放 JSON 字符串，兼容当前 `CompleteMessage.content` 设计。
- 当前项目已实现消息 `content` 的 AES-GCM 存储加密，因此文件消息元数据也会被加密存储。
- 文件本体不进入消息表，只在 `UserUploadFiles` 中保存。

### 4.9 消息类型建议

当前项目中，私聊与聊天室的 `messageType` 是业务线内部约定。建议先做简单约定：

私聊：

```text
messageType = 1 普通文本消息
messageType = 2 文件消息
```

聊天室：

```text
messageType = 0 进入聊天室
messageType = 1 普通文本消息
messageType = 2 退出聊天室
messageType = 3 文件消息
```

后端处理时，文件消息和文本消息的主要区别是：

- 文本消息：`content` 是普通文本。
- 文件消息：`content` 是文件元数据 JSON。

第一阶段可以让文件消息走和文本消息相同的落库/推送流程，只是前端根据 `messageKind=FILE` 或 `messageType` 展示成文件卡片。

## 5. 前端配合方式

### 5.1 上传文件

前端选择文件后，先调用 HTTP 上传接口：

```typescript
const formData = new FormData();
formData.append('file', file);

const response = await fetch('/file/upload', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${token}`
  },
  body: formData
});

const result = await response.json();
```

不要手动设置 `Content-Type`，浏览器会自动生成 `multipart/form-data` 的 boundary。

### 5.2 发送 WebSocket 文件消息

上传成功后，前端拿到：

```typescript
const fileInfo = result.data;
```

然后发送 WebSocket 消息：

```typescript
const fileMessage = {
  appId: 2,
  uid: currentUserId,
  toId: receiverId,
  messageType: 2,
  content: JSON.stringify({
    messageKind: 'FILE',
    fileId: fileInfo.fileId,
    fileName: fileInfo.fileName,
    fileSize: fileInfo.fileSize,
    contentType: fileInfo.contentType
  }),
  timeStamp: Date.now()
};

ws.send(JSON.stringify(fileMessage));
```

如果是聊天室文件消息：

```typescript
const roomFileMessage = {
  appId: 1,
  uid: currentUserId,
  toId: roomId,
  messageType: 3,
  content: JSON.stringify({
    messageKind: 'FILE',
    fileId: fileInfo.fileId,
    fileName: fileInfo.fileName,
    fileSize: fileInfo.fileSize,
    contentType: fileInfo.contentType
  }),
  timeStamp: Date.now()
};

ws.send(JSON.stringify(roomFileMessage));
```

### 5.3 展示文件消息

前端收到消息后：

```typescript
const payload = JSON.parse(message.content);

if (payload.messageKind === 'FILE') {
  // 展示文件卡片
}
```

文件卡片建议展示：

```text
文件名
文件大小
文件类型图标
下载按钮
```

### 5.4 下载文件

点击下载时：

```typescript
window.open(`/file/download/${payload.fileId}`, '_blank');
```

如果需要带 token，推荐使用 `fetch` 下载 Blob：

```typescript
const response = await fetch(`/file/download/${payload.fileId}`, {
  headers: {
    Authorization: `Bearer ${token}`
  }
});

const blob = await response.blob();
const url = URL.createObjectURL(blob);
const a = document.createElement('a');
a.href = url;
a.download = payload.fileName;
a.click();
URL.revokeObjectURL(url);
```

## 6. 与现有加密能力的关系

当前项目已经有：

- WSS：保护传输链路
- AES-256-GCM：保护消息 `content` 存储

文件传输功能中：

- 文件上传/download 通过 HTTP REST 走 Spring Boot，如果前端页面正式用 HTTPS，REST 接口也应通过 HTTPS。
- WebSocket 文件消息走 WSS。
- 消息表中的文件元数据 JSON 会走 AES-GCM 加密存储。
- 文件本体保存在本地磁盘，不走 AES-GCM 消息加密。

如果后续希望文件本体也加密，可以在保存磁盘前对文件流加密。但第一阶段不建议做，复杂度会明显增加。

- 增加文件选择和上传

## 8. 推荐实施顺序

1. 新建 `UserUploadFiles/` 并配置上传目录。
2. 建 `graduation_upload_file` 表。
3. 实现 `/file/upload`，先验证文件可以保存到本地磁盘。
4. 实现 `/file/download/{fileId}`，先验证上传后能下载。
5. 前端上传文件后，通过 WebSocket 发送文件消息。
6. 后端复用现有私聊/聊天室消息保存逻辑保存文件消息。
7. 前端收到文件消息后展示文件卡片。
8. 最后再补下载权限校验、文件类型白名单和大小限制。

## 9. 后续语音消息扩展

语音消息可以复用同一套文件传输能力：

```text
前端录音 -> 生成 audio/webm 或 audio/mp3 文件 -> 调用 /file/upload -> WebSocket 发送 AUDIO 消息 -> 接收方点击播放
```

也就是说，语音消息本质上也是文件消息，只是前端展示方式从“下载文件卡片”变成“音频播放器”。
