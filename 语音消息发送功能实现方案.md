# 语音消息发送功能实现方案

## 1. 目标

本文档用于设计当前聊天室项目中的语音消息发送功能。目标是用尽量简单的方式实现：

```text
前端录音 -> 得到音频文件 -> 调用后端文件上传接口 -> 拿到 fileId -> WebSocket 发送语音消息 -> 接收方播放
```

当前项目已经支持：

- WSS 传输加密
- 文件上传到本地磁盘
- `/file/upload` 上传接口
- `/file/download/{fileId}` 下载接口
- 消息 `content` 的 AES-256-GCM 存储加密

因此语音消息不需要单独设计一套二进制传输协议，可以直接复用“文件上传 + WebSocket 消息通知”的能力。

## 2. 总体方案

语音消息本质上是一种特殊文件消息：

```text
普通文件：上传任意文件 -> 发送 FILE 消息
语音消息：录制音频文件 -> 上传音频文件 -> 发送 AUDIO 消息
```

后端文件存储方式不变：

```text
UserUploadFiles/yyyy/MM/dd/uuid.webm
```

MySQL 文件元数据表不变：

```text
graduation_upload_file
```

语音消息与普通文件消息的区别主要在 WebSocket 消息类型：

```text
messageType 用于区分文本、文件、语音等消息类型
```

## 3. 消息类型约定

建议使用 `messageType` 明确区分消息类型。

### 3.1 私聊消息类型

私聊业务中建议约定：

```text
messageType = 1 文本消息
messageType = 2 文件消息
messageType = 3 语音消息
```

### 3.2 聊天室消息类型

聊天室业务中当前已有：

```text
messageType = 0 进入聊天室
messageType = 1 发送普通文本消息
messageType = 2 退出聊天室
```

建议新增：

```text
messageType = 3 文件消息
messageType = 4 语音消息
```

这样前端可以直接通过 `messageType` 判断如何展示消息：

```text
文本消息 -> 普通文本气泡
文件消息 -> 文件卡片
语音消息 -> 语音气泡/播放器
```

## 4. WebSocket 语音消息格式

语音消息仍使用现有 `CompleteMessage` 结构，不新增字段。

### 4.1 私聊语音消息

```json
{
  "appId": 2,
  "uid": 1,
  "toId": 2,
  "messageType": 3,
  "content": "{\"fileId\":10001,\"fileName\":\"voice-1715000000000.webm\",\"fileSize\":88420,\"contentType\":\"audio/webm\",\"duration\":8}",
  "timeStamp": 1715000000000
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `appId` | 私聊业务线，按当前项目约定使用 |
| `uid` | 发送者用户 ID |
| `toId` | 接收者用户 ID |
| `messageType` | 私聊语音消息使用 `3` |
| `content` | JSON 字符串，保存语音文件元数据 |
| `timeStamp` | 发送时间戳 |

### 4.2 聊天室语音消息

```json
{
  "appId": 1,
  "uid": 1,
  "toId": 10,
  "messageType": 4,
  "content": "{\"fileId\":10001,\"fileName\":\"voice-1715000000000.webm\",\"fileSize\":88420,\"contentType\":\"audio/webm\",\"duration\":8}",
  "timeStamp": 1715000000000
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `appId` | 聊天室业务线 |
| `uid` | 发送者用户 ID |
| `toId` | 聊天室 ID |
| `messageType` | 聊天室语音消息使用 `4` |
| `content` | JSON 字符串，保存语音文件元数据 |
| `timeStamp` | 发送时间戳 |

## 5. 后端实现思路

### 5.1 复用文件上传能力

语音文件上传继续使用：

```text
POST /file/upload
```

前端上传的文件类型建议为：

```text
audio/webm
```

上传成功后，后端返回：

```json
{
  "code": 200,
  "message": "上传成功",
  "data": {
    "fileId": 10001,
    "fileName": "voice-1715000000000.webm",
    "fileSize": 88420,
    "contentType": "audio/webm"
  }
}
```

### 5.2 复用文件下载能力

语音播放时继续使用：

```text
GET /file/download/{fileId}
```

前端可以把下载得到的 Blob 转成 Object URL，然后交给 `<audio>` 播放。

### 5.3 复用现有消息保存与推送逻辑

语音消息进入 WebSocket 后，本质还是一条聊天消息：

```text
content = 语音文件元数据 JSON 字符串
```

当前项目已经实现消息 `content` 入库前 AES-GCM 加密，历史查询时解密。因此语音消息的 `content` 也会被加密保存。

数据库中消息表保存的是加密后的语音元数据，不是音频文件本体：

```text
ENC:v1:<ivBase64>:<ciphertextAndTagBase64>
```

解密后才是：

```json
{
  "fileId": 10001,
  "fileName": "voice-1715000000000.webm",
  "fileSize": 88420,
  "contentType": "audio/webm",
  "duration": 8
}
```

### 5.4 聊天室处理建议

当前 `ChatRoomProcessor` 中：

```text
messageType = 1
```

用于发送聊天室文本消息。

建议后端把 `messageType = 4` 语音消息走和文本消息同样的广播/落库流程，只是 messageType 保持为 `4`。

简单实现方式：

```text
case 1:
case 3:
case 4:
    sendChatRoomMessage(ctx, msg);
    break;
```

其中：

```text
1 = 文本
3 = 文件
4 = 语音
```

广播时不要强制把 `messageType` 写死成 `1`，应保留原始 `msg.getMessageType()`。

### 5.5 私聊处理建议

`PrivateChatProcessor` 当前不强依赖具体 `messageType`，只要 `uid`、`toId`、`content` 不为空就会保存并推送。

因此私聊语音消息可以直接使用：

```text
messageType = 3
```

后端保存和推送逻辑基本不用改。

## 6. 前端实现思路

### 6.1 Chrome 录音方式

Chrome 浏览器推荐使用原生 API：

```text
navigator.mediaDevices.getUserMedia
MediaRecorder
```

录音流程：

```text
1. 用户点击或按住录音按钮
2. 前端请求麦克风权限
3. 创建 MediaRecorder
4. recorder.start() 开始录音
5. ondataavailable 收集音频片段
6. recorder.stop() 停止录音
7. 合成 Blob
8. 转成 File
9. 调用 /file/upload 上传
10. 上传成功后发送 WebSocket 语音消息
```

### 6.2 录音伪代码

```typescript
let recorder: MediaRecorder | null = null;
let audioChunks: BlobPart[] = [];
let startTime = 0;
let currentStream: MediaStream | null = null;

async function startRecord() {
  currentStream = await navigator.mediaDevices.getUserMedia({ audio: true });

  const mimeType = MediaRecorder.isTypeSupported('audio/webm;codecs=opus')
    ? 'audio/webm;codecs=opus'
    : 'audio/webm';

  recorder = new MediaRecorder(currentStream, { mimeType });
  audioChunks = [];
  startTime = Date.now();

  recorder.ondataavailable = (event) => {
    if (event.data && event.data.size > 0) {
      audioChunks.push(event.data);
    }
  };

  recorder.start();
}

async function stopRecordAndSend(receiverId: number) {
  if (!recorder) {
    return;
  }

  recorder.onstop = async () => {
    const duration = Math.ceil((Date.now() - startTime) / 1000);
    const blob = new Blob(audioChunks, { type: recorder?.mimeType || 'audio/webm' });

    currentStream?.getTracks().forEach(track => track.stop());
    currentStream = null;

    if (duration < 1 || blob.size === 0) {
      return;
    }

    const file = new File([blob], `voice-${Date.now()}.webm`, {
      type: blob.type || 'audio/webm'
    });

    const uploadResult = await uploadFile(file);
    sendPrivateAudioMessage(receiverId, uploadResult, duration);
  };

  recorder.stop();
}
```

### 6.3 上传语音文件

复用文件上传接口：

```typescript
async function uploadFile(file: File) {
  const formData = new FormData();
  formData.append('file', file);

  const response = await fetch('/file/upload', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${token}`
    },
    body: formData
  });

  const result = await response.json();
  return result.data;
}
```

注意：上传 `FormData` 时不要手动设置 `Content-Type`，浏览器会自动带 boundary。

### 6.4 发送私聊语音消息

```typescript
function sendPrivateAudioMessage(receiverId: number, fileInfo: any, duration: number) {
  const message = {
    appId: 2,
    uid: currentUserId,
    toId: receiverId,
    messageType: 3,
    content: JSON.stringify({
      fileId: fileInfo.fileId,
      fileName: fileInfo.fileName,
      fileSize: fileInfo.fileSize,
      contentType: fileInfo.contentType,
      duration
    }),
    timeStamp: Date.now()
  };

  ws.send(JSON.stringify(message));
}
```

### 6.5 发送聊天室语音消息

```typescript
function sendChatRoomAudioMessage(roomId: number, fileInfo: any, duration: number) {
  const message = {
    appId: 1,
    uid: currentUserId,
    toId: roomId,
    messageType: 4,
    content: JSON.stringify({
      fileId: fileInfo.fileId,
      fileName: fileInfo.fileName,
      fileSize: fileInfo.fileSize,
      contentType: fileInfo.contentType,
      duration
    }),
    timeStamp: Date.now()
  };

  ws.send(JSON.stringify(message));
}
```

### 6.6 展示语音消息

前端收到消息后，需要根据 `appId + messageType` 联合判断消息类型。

注意：不能只根据 `messageType` 判断，因为不同业务线下的 `messageType` 含义不同。例如：

```text
appId = 2, messageType = 3 表示私聊语音消息
appId = 1, messageType = 3 表示聊天室文件消息
appId = 1, messageType = 4 表示聊天室语音消息
```

因此前端建议这样判断：

```typescript
function handleMessage(message: CompleteMessage) {
  if (message.appId === 2 && message.messageType === 3) {
    // 私聊语音消息
    const audioInfo = JSON.parse(message.content);
    renderAudioBubble(audioInfo);
  }

  if (message.messageType === 4 && message.appId === 1) {
    // 聊天室语音消息
    const audioInfo = JSON.parse(message.content);
    renderAudioBubble(audioInfo);
  }
}
```

语音气泡建议展示：

```text
▶ 8''
```

也可以展示：

```text
▶ 8''  voice-xxx.webm
```

如果前端还要处理普通文件消息，也建议同样使用 `appId + messageType`：

```typescript
if (message.appId === 2 && message.messageType === 2) {
  // 私聊文件消息
}

if (message.appId === 1 && message.messageType === 3) {
  // 聊天室文件消息
}
```

### 6.7 播放语音

如果下载接口需要 token，推荐用 `fetch` 获取 Blob，而不是直接 `<audio src="/file/download/10001">`。

```typescript
async function playAudio(fileId: number) {
  const response = await fetch(`/file/download/${fileId}`, {
    headers: {
      Authorization: `Bearer ${token}`
    }
  });

  const blob = await response.blob();
  const audioUrl = URL.createObjectURL(blob);
  const audio = new Audio(audioUrl);

  audio.onended = () => {
    URL.revokeObjectURL(audioUrl);
  };

  await audio.play();
}
```

如果要用 `<audio controls>`，也可以先把 Blob URL 设置给组件：

```typescript
audioElement.src = audioUrl;
```

## 7. 前端注意事项

### 7.1 麦克风权限

浏览器会弹出麦克风授权。用户拒绝时，需要提示：

```text
请允许浏览器访问麦克风后再发送语音
```

### 7.2 HTTPS / localhost 限制

`getUserMedia` 通常要求安全上下文：

```text
https:// 页面
localhost 页面
```

本地开发使用 Chrome + `localhost` 通常可以正常录音。

### 7.3 录音格式

Chrome 推荐：

```text
audio/webm;codecs=opus
```

如果不支持，再降级：

```text
audio/webm
```

### 7.4 时长限制

建议前端限制：

```text
最短：1 秒
最长：60 秒
```

超过 60 秒自动停止录音并上传。

### 7.5 文件大小限制

当前后端文件上传默认限制：

```text
10MB
```

语音消息通常远小于 10MB。如果后续希望更严格，可以前端限制语音文件大小，例如：

```text
最大 5MB
```

除了业务层 `FileService` 的大小限制外，Spring Boot 自身也有 multipart 上传大小限制。为了避免音频上传时被 Spring 在进入业务代码前拦截，建议在 `application.yml` 中配置：

```yaml
spring:
  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 10MB
```

这两个值建议和后端 `file.upload.max-size-mb` 保持一致：

```yaml
file:
  upload:
    base-dir: UserUploadFiles
    max-size-mb: 10
```

### 7.6 录音结束后释放麦克风

停止录音后一定要释放：

```typescript
currentStream?.getTracks().forEach(track => track.stop());
```

否则浏览器可能一直显示麦克风正在使用。

## 8. 后端是否还需要改

第一阶段可以尽量少改后端。

必须确认：

- `/file/upload` 可以上传 `audio/webm`
- `/file/download/{fileId}` 可以下载音频文件
- 私聊消息能保存并推送 `messageType=3`
- 聊天室消息能保存并广播 `messageType=4`
- 聊天室广播时保留原始 `messageType`，不要把文件/语音消息改成文本消息

后端建议做的小改动：

1. `ChatRoomProcessor` 支持 `messageType=3` 文件消息和 `messageType=4` 语音消息。
2. 广播聊天室消息时保留原始 `messageType`，不要写死成 `1`。

当前后端如果已经使用 `MessageTypeEnum`，建议枚举约定保持为：

```text
私聊：
messageType = 1 文本消息
messageType = 2 文件消息
messageType = 3 语音消息

聊天室：
messageType = 0 进入聊天室
messageType = 1 文本消息
messageType = 2 退出聊天室
messageType = 3 文件消息
messageType = 4 语音消息
```

私聊目前一般不需要额外改动。

## 9. 推荐实施顺序

1. 前端先实现录音，确认可以得到 `audio/webm` Blob。
2. 前端把 Blob 转成 File，调用 `/file/upload`。
3. 前端上传成功后发送 `messageType=3` 的私聊语音消息。
4. 接收方收到后展示语音气泡。
5. 点击语音气泡，通过 `/file/download/{fileId}` 获取 Blob 并播放。
6. 再补聊天室语音消息，使用 `messageType=4`。
7. 后端如果聊天室暂时不支持 `messageType=4`，再补对应分支。

## 10. 总结

语音发送最简单的实现方式是：

```text
语音 = 音频文件 + 特定 messageType 的聊天消息
```

后端继续负责：

```text
保存音频文件
返回 fileId
保存/推送聊天消息
下载音频文件
```

前端负责：

```text
录音
上传音频
发送语音消息
展示和播放语音
```

这种方案不需要 WebSocket 传二进制，也不需要后端处理音频编码，适合当前项目快速落地。
